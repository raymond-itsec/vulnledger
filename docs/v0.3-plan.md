# v0.3 Plan (technical)

The full engineering plan for the v0.3.0 multi-host transition. For the non-technical version, see [Roadmap](Roadmap.md).

## Hypervisor decision: Multipass

| | Multipass | VMWare Workstation Pro | KVM + libvirt direct |
|---|---|---|---|
| License | Apache 2.0, free | Free for personal, license needed for commercial | Free, open source |
| Headless | Yes, native | Desktop-oriented | Yes |
| cloud-init | First-class | Extra plumbing required | Possible via libvirt-image-builder |
| Performance | KVM/qemu backend on Linux = native | KVM-equivalent, slight wrapper overhead | Native, no wrapper |
| Snapshots / clones | Yes | Yes (more polished) | Yes |
| Already installed | Yes | No | Available on Ubuntu |
| Setup friction | None | Install + license check | Most setup work |

**Decision: Multipass with KVM/qemu driver.** Already installed on the dev box, free, headless-native, cloud-init-ready, performance equivalent to direct KVM. VMWare adds licensing/install friction for no functional advantage. Direct KVM is more powerful but you'd be writing tooling Multipass already provides.

```bash
sudo multipass set local.driver=qemu
```

## Architecture commitments locked in

| Decision | Resolution |
|---|---|
| Hypervisor | Multipass + KVM/qemu |
| Repo layout | `deploy/` subdirectory in main repo |
| Compose split file naming | `deploy/compose/{edge,app,data,monitoring}.yml` |
| Env file naming | `deploy/profiles/{allinone,multihost}.env` |
| Operating modes | `make allinone-up` (single-host dev) and `make multihost-up` (4 VMs) - same source files |
| Service discovery | Env-driven hostnames (`data.vl.local`, `app.vl.local`, etc.) |
| Public Grafana dashboard URL | `/status/` proxied via Caddy |
| Internal Grafana dashboard URL | `/grafana/` proxied via Caddy, SSO required |
| API URL versioning | New `/api/v1/` prefix; old `/api/...` becomes 301 → v1 alias for transition |
| Observability data flow | backend `/metrics` → vmagent → VictoriaMetrics → Grafana / vmalert → Alertmanager → Slack |
| WireGuard subnet | `10.200.0.0/24` |
| Off-host backup | Parked - manual `tar` of `seaweedfs_data` for now |
| Schema rename | Bundle into v0.3.0 (`Clients` → `Customers`, `ReviewSessions` → `Projects`) |

### Phase 11: Safe retries on POST creates (replay-safe writes)

| Step | Scope |
|---|---|
| 11.1 | New `request_replay_keys` table (key, request_hash, response_body, expires_at). Stores the result of each first-seen request so a retry returns the same answer instead of creating a duplicate. |
| 11.2 | Middleware on POST routes: client sends a key in a header; if seen before, return cached response; if new, run the handler and cache the result for the key's lifetime. |
| 11.3 | Documented in API reference. |

## Suggested execution order

If working solo over multiple sessions, this order minimizes rework:

1. Phase 1 audits + small commits (1.1-1.7) - low risk, sets foundation
2. Phase 2 compose refactor - biggest single commit, validates against single host
3. Phase 3-6 multipass + WG + per-tier deploy + segmentation - the multi-host validation
4. Phase 7 schema rename - invasive but bounded; test in multi-host setup
5. Phase 8 observability - biggest payoff for ops, can run in parallel with Phase 9
6. Phase 9 container registry - depends on Phase 2 compose split being stable
7. Phase 10 soft delete + Phase 11 safe retries - independent, can interleave
8. Phase 1.8 settings DI - slot in when you have a quiet session
9. Phase 12 validation drill - the gate before tagging
10. Phase 13 release
